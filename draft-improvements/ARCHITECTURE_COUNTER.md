# Architecture

This repository is an AI coding harness.

Has to be readable for both software engineers and machines.

A smaller file is only better when it clarifies
ownership, reduces context load, and makes validation more local.

We divide by domain to clarify boundaries and we apply hexagonal architecture to increase the isolation.

We try to maintain agent context as small and isolated as possible.

## Purpose and core ideas of this repo

We are building a portable tool (as plug and play as possible) that automates the full software engineering loop release lifecycle.

In order to do that we use building a harness for AI.

The idea is we run a pipeline that orchestates steps composed by deterministic code and by AI focused workers in charge of the indetermisistic steps.

We try the pipelines to be as reproducible as possible so each AI indeterminisic step has to have a single clear task.

Harness will act as an orchestator and will guide along each step of the pipelines.

### The lifecycle of the SE

Main phases of our SE lifecycle are based on the current SDD lierature:

- `EXPLORE`, `PROPOSAL`, `SPEC`, `DESIGN`, `TASKS`, `IMPLEMENT`, `TEST`, `REVIEW`, `ARCHIVE`

But we do some modifications.

We perform what we call a `TDD_LOOP` at the end instead implement - test. 

TDD loop loops over tdd standards

- Create required test code. Test that are going to fail but describes the expected behaviur.
- Create the code that makes that test works.
- Review checks all criteria are right: All test for the described behaviur. All code for test. All test green...
- Iterate until reviewer gives the approval.

Also we extract learning at the end of EXPLORER phases and at the end of TDD_LOOP. 

So real harness lifecycle simplified steps is more like:

- `EXPLORE`, `KNOWLEADGE_PHASE_EXTRACTOR`, `PROPOSAL`, `SPEC`, `DESIGN`, `TASKS`, `TDD_LOOP`, `KNOWLEADGE_PHASE_EXTRACTOR`

Also any phase can escalate some problem to previus phases or ask user if something is unclear. Ex: Design says not touch some file but implementer realised is required. Or found unexpected bug not described. Then escalation process should decide how to resolve.

Appart from that release lifecycle follows git trunk based development and its powered by the artifact generated by github/gitlab CI features.

So real full lifecycle will be:

- There is a main branch with generated artifacts that contains information repository status. 
- We create our feature branch.
- We start the full SDD lifecycle. 
- EXPLORER phase will use the master (or branch latest) artifacts if exist. Will extract relevant information from git CI output artifacts about repository status.
- Next phases of SDD will use that info to take better decisions.
- At the end of SDD our code is pushed to feature branch. 
- Push to feature triggers feature CI steps.
- MR to master is created, reviewed and approved manually. Has to pass the CI branch steps. 
- When MR is merged to master triggers CI pipeline will create new master artifacts.
- We repeat the loop.

So actually real harness pipeline is composed by steps performed by SDD + trunk based. One side lives in github/gitlab other runs locally inside a computer.

### The learning phases purpose and expectations

Problem learning phases are trying to solve is not about to create a knowledge base like jira.

What we want is a regenerative database with information about the folders is about to touch. This will help our agents to complete the tasks spending less tokens and having less iterations to get the relevant information. Also our expectation is to find also a way to generate humans files from this learning system. Not just fast access data for agents.

We want to store in our learning system relevant information about repositories, code, flows, current behaviuor... Relevant things that will help agents map faster the info they looking for.

Also we want this to be regenerative. If you delete the DB should be capable to start fresh again and eventually get into to the same position.

Appart from that future expectation is we can somehow extract that DB structure prepared specially for machines and turn it into more humman redable files. Like mds describing the domains, flows, interactions, ideas behind each part of code, diagrams...


#### Learning pieces

Learning pieces would be the follow:

- Source of true (SOT): Lives in git files.
- Local agents DB (LADB): From source of true we create local copy DB for agents.
- Humman knowleadge files(HKF): From source of true we create remote readable files.
- Knowleadge patches (KP): Data learn by our SDD flow during `KNOWLEADGE_PHASE_EXTRACTOR`
- KP to SOT pipeline: Flow that allows automatically merge the patches into source of true.
- SOT to LADB script: Script that erases and populates again from SOT to LADB.
- SOT to HKF pipeline: Flow that allows us to generate HKF from SOT.
